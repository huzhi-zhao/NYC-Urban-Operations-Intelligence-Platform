BO-1 · Service Request Load Monitoring
How overloaded is each borough's city services on any given day?
P0 Core 311 → Silver
Real-world precedent
Chicago WindyGrid · City of Chicago · 2012–2018
Integrated 311, 911, weather, and infrastructure data into a single ops dashboard. Dispatchers could see complaint hotspots by ward in real time and pre-position crews. Reduced average 311 response time by 18% in the first year. Decommissioned in 2018 when replaced by the citywide Salesforce platform — but the architecture is identical to what we are building.

BO-2 · Public Safety Incident Tracking
Where and when do traffic collisions and shooting incidents concentrate by borough?
P0 Core NYPD → Silver
Real-world precedent
NYPD Domain Awareness System · Microsoft + NYPD · 2012–present
Correlates collision data, crime complaints, and shooting incidents with patrol zone boundaries. Used to redeploy traffic enforcement units to high-collision corridors. The collision dataset (resource ID h9gi-nx95) is the exact same Socrata feed we ingest. The DAS ingests it in real time; we ingest it as daily backfill — different latency, same source of truth.

BO-3 · Weather-Operations Correlation
Does extreme weather predict service demand spikes? Can we pre-position resources before a storm?
P1 Enrichment Open-Meteo → Silver
Real-world precedent
NYC DSNY LoadIQ · Dept of Sanitation · internal project · ~2021
Correlates snowfall forecasts with 311 snow-removal complaints to pre-position salt spreaders by district the night before a storm. Reduced overtime cost by approximately $2M in winter 2022. Uses the same NWS/Open-Meteo feed we ingest. This is the single closest precedent to our BO-3 + BO-5 combination.

BO-4 · Borough Spatial Enrichment
Assign every event to a canonical borough via spatial join — never trust the raw text field alone
P0 Enabler DCP → dim_geography
Real-world precedent
LA Sanitation + Palantir Foundry · 2019
Raw 311 records had district fields filled inconsistently by call-center operators — 30% mismatch rate between text field and GPS coordinates. Palantir rebuilt the spatial join from GPS to census tracts, which became the authoritative borough assignment. NYC 311 has the same problem: the "borough" text field is null or wrong in a significant fraction of records. GPS → polygon is the correct approach.
Quality Gate
Spatial join hit rate
> 90% of records with lat/lon
Fallback strategy
Text borough field (flagged as degraded)
Unresolvable
NULL borough_id — excluded from Load Score

BO-5 · Operational Load Score
A single daily score per borough that drives resource allocation recommendations
P0 Final Output Gold Layer
Real-world precedent
NYC Mayor's Office of Operations · Citywide Performance Reporting (CPR) · ongoing
Publishes a composite agency-performance score per district each quarter, fed directly into budget and headcount allocation decisions. Our Load Score is the real-time daily equivalent — same concept, 90× higher cadence. CPR uses 311 response time and complaint volume as primary inputs, which are our BO-1 inputs.
Score formula (draft — to be confirmed)
Component A (40%)
311 weighted complaint volume (complaint_weight × count)
Component B (40%)
NYPD collision severity (injuries×1 + fatalities×5)
Component C (20%)
Weather severity score multiplier
